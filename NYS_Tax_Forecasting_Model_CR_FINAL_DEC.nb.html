This last section of work was dedicated to refining the original models and experimenting with potential optimization through the development of alternative models.

Linear Model

Optimization for the linear regression model proved to be difficult on the basis that the original model proved to be most accurate. This testing included the following linear regression models:

1.variance inflation factor (VIF) because of the close scoring for CPI and GDP. Ultimately, the removal of any of these attributes worsened the performance of the original model and their inclusion was necessary for capture general trends.

  1. a model using the caret package with support for Ridge regression, but this method of optimization proved to be ineffective likely due to the small size of our dataset. There was no true worry about over fitting because the data did not heavily fluctuate nor was there enough data to warrant a coefficient shrinking, which is typical to the Ridge-Regression Approach.

  2. A model using standard caret (the caret package is a useful tool for optimizing models in R, and is managed by CRAN) hyptertuning which reinforced the parameters of the original model.

The inclusion of all 4 models (the original included) is to show sufficient effort in attempting to tune the model further.

RF Model

The optimized Random forest model that came with great success. There are only two models to demonstrate, the original and the optimized hypertuned model. The hypertuned model used the ranger package (also managed by CRAN) in conjunction with caret to find the ideal parameters for the mtry, min.node.size. and splitrule parameters. These parameters are provided upon running the model. This was the most accurate model within a 2.99% error.

ETS Model

There was not inherent caret support was this model is based off the forecasting package that employs a different technical methodology than that of the linear regression and random forest models. I mimicked the caret functionality by having the ETS choose from a menu of available parameters. Ultimately, this model reinforced the findings of the original model by selecting the same parameters.

# This offset is necessary because the NYS API for tax pulls has a limit of 1000 lines. 
# The tax data has approximately 1,600 lines. This method is recommended by the NYS data contractor, and it doesn't require an API key!

fetch_data <- function(offset, limit = 1000) {
  url <- paste0(base_url, "?$limit=", limit, "&$offset=", offset)
  response <- GET(url)
  
  if (http_type(response) != "application/json") {
    stop("Failed to fetch data: ", status_code(response))
  }
  
  content <- content(response, as = "text", encoding = "UTF-8")
  data <- fromJSON(content, flatten = TRUE)  # flatten nested data for simplicity
  
  if (is.null(data) || length(data) == 0) {
    return(NULL)  # Return NULL if data is empty
  }
  
  return(as.data.frame(data))
}
